# SCALING-ROADMAP — АгентСфера

> План масштабирования: от одного агента к SaaS-продукту для индивидуальных страховых агентов.
> Каждый этап — самостоятельная ценность. Не нужно делать всё сразу.

---

## ТЕКУЩЕЕ СОСТОЯНИЕ

**Что уже есть:**
- Полный CRM-функционал (клиенты, договоры, взносы, комиссии, задачи, календарь, акты, дашборд)
- Изоляция данных по userId — каждый агент видит только своё
- Регистрация и авторизация (JWT + refresh token + автообновление)
- Мобильная адаптация + PWA
- Импорт/экспорт Excel
- AI-парсинг актов (Anthropic Claude)
- VPS с MongoDB (авторизация + localhost + фаервол), nginx, PM2

**Что нужно для продажи:**
- Система тарифов и оплаты
- Верификация пользователей
- Юридическая база
- Админ-панель
- Мониторинг и стабильность

## ОЦЕНКА ПРОИЗВОДИТЕЛЬНОСТИ

### Текущая ёмкость (без изменений кода)

| Параметр | Лимит | Комментарий |
|----------|-------|-------------|
| Одновременных пользователей | ~100 | Node.js single-thread + PM2 single instance |
| Договоров в базе | ~50 000 | Индексы на месте, пагинация есть |
| Клиентов в базе | ~20 000 | Поиск по индексу userId+name |
| Запросов в минуту | ~1000 | Rate limiter + nginx |
| Размер базы данных | ~2-5 GB | 60GB NVMe — запас 10x |
| Размер билда (клиент) | 112KB gzip | Быстрая загрузка даже на мобильном |

### Что нужно оптимизировать (точечно, не переписывая)

| Проблема | Когда станет заметно | Решение | Трудозатраты |
|----------|---------------------|---------|-------------|
| getTotals: find + цикл в JS | 5000+ договоров | MongoDB aggregation ($group, $sum) | 1 час, 30 строк |
| dashboardService: 4× дублирование Contract.find (см. детали ниже) | 1000+ договоров | Один запрос → передать во все функции, или объединённый endpoint | 1 час |
| Календарь: загружает ВСЕ события | 1000+ задач | Серверный endpoint /calendar/events?month= | 2 часа, новый роут |
| PM2 single instance | 100+ одновременных | `pm2 start app.js -i 2` (cluster mode) | 1 минута, одна команда |
| Нет кеширования дашборда | 200+ пользователей | Redis кеш с инвалидацией | 4 часа |
| Rate limiting in-memory | Несколько инстансов | Redis store для rate limiter | 1 час |

### Что НЕ нужно менять

| Компонент | Почему не трогаем |
|-----------|------------------|
| Express.js | Стандартный стек для SaaS, масштабируется горизонтально |
| MongoDB | Отлично подходит для CRM, масштабируется через Atlas/реплики |
| React (Vite) | 112KB билд — легче чем 90% SaaS-продуктов |
| Mongoose ODM | Валидация, виртуальные поля, индексы — всё используется |
| JWT авторизация | Stateless — работает при нескольких инстансах без Redis |
| Структура проекта | models→services→routes — чистое разделение, расширяется легко |
| CSS без фреймворков | Легче на 200KB+ чем MUI/Ant, уникальный дизайн |

### Пороги масштабирования

```
1-50 агентов      → Текущий код работает без изменений
50-200 агентов    → PM2 cluster mode + aggregation в getTotals/dashboard
200-500 агентов   → Redis кеширование + серверный календарь
500-1000 агентов  → MongoDB Atlas + CDN (Cloudflare) + больший VPS
1000+ агентов     → Несколько серверов + балансировщик + Redis sessions
```

## ДЕТАЛЬНЫЙ АУДИТ КОДА (23.03.2026)

### Серверные проблемы (влияют на производительность)

**Проблема 1: dashboardService — 4 дублирующихся запроса Contract.find**

При открытии дашборда клиент вызывает 5 API-запросов параллельно (Promise.all). На сервере каждый из `getStats`, `getAlerts`, `getByCompany`, `getByType` делает свой отдельный `Contract.find({ userId })`. Это 4 одинаковых запроса к MongoDB за теми же данными.

- Файл: `server/src/services/dashboardService.js`
- Влияние: при 500 договорах — 4× загрузка 500 документов вместо 1×
- Решение: один `Contract.find({ userId })` + передача результата во все функции, или объединённый endpoint `GET /dashboard/all`
- Трудозатраты: 1 час

**Проблема 2: getByCompany и getByType — почти идентичный код**

Обе функции делают `Contract.find → forEach → группировка по полю → sort`. Единственное отличие — ключ группировки (`company` vs `type`). ~60 строк дублированного кода.

- Файл: `server/src/services/dashboardService.js`
- Влияние: нет на производительность, но лишний код
- Решение: одна общая функция `groupContractsBy(contracts, field)`
- Трудозатраты: 15 мин

**Проблема 3: toJSON() для доступа к commissionAmount**

В dashboardService везде используется `c.toJSON().commissionAmount` — потому что виртуальное поле commissionAmount не доступно на объекте документа напрямую, только при сериализации. Каждый вызов `toJSON()` сериализует ВЕСЬ документ (все поля, все взносы) ради одного числа.

- Файлы: `server/src/services/dashboardService.js`, `server/src/services/contractService.js` (getTotals)
- Влияние: микрооптимизация, заметно при >5000 договоров
- Решение: вычислять инлайн: `c.commissionType === '%' ? Math.round(c.premium * c.commissionValue / 100) : c.commissionValue`
- Трудозатраты: 15 мин

### Клиентские проблемы (качество кода, не производительность)

**Проблема 4: SVG-иконки дублируются между страницами (~150 строк)**

Одни и те же SVG-иконки (IconChevron, иконки редактирования, удаления, поиска, календаря) заново определяются как React-компоненты в каждой странице: Dashboard.jsx (~10 иконок), ClientDetail.jsx, Contracts.jsx, Tasks.jsx, Calendar.jsx, Acts.jsx. Суммарно ~150 строк дублированного SVG-кода.

- Файлы: все JSX-файлы в `client/src/pages/`
- Влияние: нет на производительность (Vite tree-shakes), но затрудняет поддержку — при изменении иконки нужно менять в 6 местах
- Решение: вынести общие иконки в `client/src/components/Icons.jsx` и импортировать
- Трудозатраты: 30 мин

**Проблема 5: formatDate/formatCurrency дублируются (~40 строк)**

Каждая страница определяет свою версию утилит форматирования:
- Dashboard.jsx: `formatMoney()`, `formatDate()`, `formatDateShort()`
- ClientDetail.jsx: `formatCurrency()`, `formatDate()`, `formatBirthday()`
- Contracts.jsx: `formatCurrency()`, `formatDate()`
- Tasks.jsx, Calendar.jsx, Acts.jsx — свои копии

При этом форматы слегка отличаются (где-то `2-digit` год, где-то `numeric`), что может приводить к визуальным несоответствиям между страницами.

- Файлы: все JSX-файлы в `client/src/pages/`
- Влияние: нет на производительность, но несогласованное форматирование дат/чисел
- Решение: вынести в `client/src/utils/format.js` с единым API: `formatMoney(val)`, `formatDate(date, style)`, `formatDateShort(date)`
- Трудозатраты: 20 мин

### Что НЕ является проблемой (выглядит большим, но нужно)

| Файл | Размер | Почему не проблема |
|------|--------|-------------------|
| Dashboard.jsx (23KB) | 50% — SVG-иконки | Реальная логика ~12KB — 5 секций дашборда, адекватно |
| ClientDetail.jsx (33KB) | 5 секций: шапка, договоры, заметки, задачи, модалки | Можно разбить на под-компоненты для читаемости, но кода не станет меньше |
| ContractForm.jsx (29KB) | 3-шаговая форма с объектами авто/недвижимость/жизнь | Сложная форма, размер оправдан |
| 5 API-запросов дашборда | Promise.all параллельно | Клиентский код правильный, проблема на сервере (дублирование find) |
| CSS-файлы (8 штук, ~88KB суммарно) | Каждая страница свой CSS | Нет конфликтов, чистое разделение, CSS-переменные |

## ЭТАП 1 — ГОТОВНОСТЬ К ПЕРВЫМ ПОЛЬЗОВАТЕЛЯМ

> Цель: 10-50 агентов могут безопасно пользоваться системой
> Срок: 1-2 недели

### 1.1 Подтверждение email при регистрации
- Отправка кода/ссылки на email после регистрации
- Аккаунт неактивен до подтверждения
- Пакет: nodemailer + SMTP (Яндекс, Mail.ru или SendGrid)
- Файлы: `server/src/services/emailService.js`, обновить `routes/auth.js`, модель User (+emailVerified, +verificationToken)

### 1.2 Восстановление пароля
- «Забыли пароль?» на странице логина
- Email с ссылкой/кодом для сброса
- Файлы: `routes/auth.js` (POST /auth/forgot, POST /auth/reset), обновить Login.jsx

### 1.3 Автотесты бэкенда (задача 6.5)
- Jest + Supertest + mongodb-memory-server
- 5 групп: авторизация, клиенты, договоры, акты, каскадное удаление
- ~15-20 тестов, ~200 строк
- Файлы: `server/tests/`

### 1.4 Бэкапы MongoDB
- Автоматический mongodump по cron (ежедневно, хранить 30 дней)
- Скрипт: `server/scripts/backup.sh`
- Cron: `0 3 * * * /var/www/agentsfera/server/scripts/backup.sh`

### 1.5 SSH безопасность
- Пользователь `deploy` вместо root
- SSH по ключу, отключить вход по паролю
- PermitRootLogin no

### 1.6 Мониторинг
- UptimeRobot (бесплатно) — проверяет доступность каждые 5 мин, уведомления в Telegram
- PM2 Plus или pm2-logrotate для ротации логов

### 1.7 Юридика (минимум для старта)

- Кеширование дашборда (инвалидация при изменении данных)

### 4.2 Миграция MongoDB

- Или реплика-сет на отдельном сервере

### 4.3 CDN для статики
- Cloudflare (бесплатный тариф) — кеширование статики, защита от DDoS
- Или Selectel CDN для РФ

### 4.4 CI/CD
- GitHub Actions: автотесты при push, автодеплой при merge в main

### 4.5 Логирование и мониторинг
- Grafana + Prometheus или Sentry для ошибок
- Алерты при росте ошибок, падении сервера

### 4.6 Горизонтальное масштабирование
- Несколько VPS за балансировщиком (если >1000 пользователей одновременно)
- Stateless архитектура (токены в Redis, файлы в S3)

## ЭТАП 5 — АГЕНТСТВА (B2B)

> Цель: продажа не только агентам, но и страховым агентствам
> Срок: когда будет product-market fit с агентами

### 5.1 Мультитенантность
- Модель Organization (агентство)
- Агент привязан к организации
- Руководитель видит данные всех агентов организации
- Дашборд агентства: общая аналитика, сравнение агентов

### 5.2 Роли внутри организации
- Владелец (полный доступ)
- Менеджер (видит всех агентов, не может удалять)
- Агент (видит только своё)

### 5.3 Отчёты для руководителя
- Сводка по агентам: кто сколько собрал, КВ, задолженность
- Экспорт отчётов в Excel/PDF
- Графики: динамика по месяцам, сравнение агентов

### 5.4 Тарифы для агентств
- Цена за агента (например 300₽/мес за агента)
- Скидка за объём (10+ агентов — дешевле)
